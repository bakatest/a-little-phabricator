Prelude

Phabricator 是 Facebook 开源的团队协作工具。

Phabricator 的模型较为灵活，为进一步促进有效的工作沟通、减少不必要的打扰，需要对 Phabricator 的使用方法进行约定。

Project 是什么

功能上看似杂乱，但 project 其实只有三个作用

  • 信息标签 - 简化内容展示、方便分类和检索
  • 邮件列表 - 订阅邮件、方便邮件协作
  • 策略对象 - 应用于「members of projects」策略

策略对象

Project vs. Repository

Q1：代码仓库一定有独立的 Project 么？

A1：不一定。代码仓库可以共用一个 Project Tag，比如 #LDAP 可以同时标记在 ldap-admin-backend 和 ldap-admin-web。

Q2：代码仓库可以有多个 Project Tag 么？

A2：是的。比如仓库 ldap-admin-web 可同时标记为 #LDAP 和 #Infrastructure。

Project vs. Repository

Q3：代码仓库可以没有 Project 么？

A3：可以。代码仓库不使用邮件订阅、任务管理功能时应避免创建。

Q4：代码仓库必须加上 Group Tag 么？

A4：待定。有好处也有坏处，目前的做法是推荐加上 Group Tag。比如在 ldap-admin-web 上的 #DevX (产品组)。

PS：添加 Timeline Tag 到一个仓库的意义是不明确的。

Policy

对象策略 (Policy)

这里对象策略和上文策略对象是两个不同的概念。

包括任务、Project、文档、代码仓库在内的 Phabricator 对象都通过 Policy 机制实现访问控制（注意！Project Tag 不影响访问权限）。

Repository 对象策略

代码仓库的对象策略无法通过表单自动配置

创建代码仓库之后从左侧面板进入「Policies」页面可以 review 代码仓库的对象策略，完成策略的调整之后再 activate 代码仓库。

Visible to Editable by Pushable by
  • Users @<user>...
  • Members of #<group>
  • Users @<user>...
  • Administrators
  • All Users
